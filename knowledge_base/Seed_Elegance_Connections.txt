THE MASTER LIST: SEED-ELEGANCE CONNECTIONS
The Meta-Framework of Universal Problem-Solving Archetypes
By Anand, Advaitian Foundation

========================================================
OVERVIEW
========================================================

This is the meta-framework for all 1000 commentaries. These are the universal problem-solving archetypes—the structural patterns that recur across all problems. Every problem is a disguised instance of one (or more) of these deep connections.

IMPORTANT NOTE: One particular problem can be solved using multiple different approaches (which would fall into one or more of these archetypes). The "Connections" section in every commentary must capture BOTH:
1. Application of "identified archetypes" to other topics/concepts
2. Other archetypes applicable to the current problem

The 20 archetypes are organized into FIVE META-CATEGORIES:
- Category 1: Structure Recognition (see the hidden architecture)
- Category 2: Transformation Thinking (change representation to reveal simplicity)
- Category 3: Constraint Exploitation (the restrictions ARE the solution)
- Category 4: Counting & Extremization (structure the chaos)
- Category 5: Meta-Reasoning (think about the problem structure itself)

THE META-META PRINCIPLE:
"Every problem is a seed-elegance connection waiting to be named."

When you encounter a new problem:
1. Ask: Which of these 20 archetypes does this resemble?
2. Name the seed: What is the underlying structure?
3. Find the pivot: Which archetype unlocks it?
4. Extract the takeaway: What meta-principle emerges?

========================================================
CATEGORY 1: STRUCTURE RECOGNITION
"See the hidden architecture."
========================================================

1. INVARIANCE
Seed: Something doesn't change, even as everything else does.
Elegance: Track the invariant instead of tracking all moving parts.
Examples:
- Sum of angles in a polygon = (n-2)×180° [Commentary #1]
- Conservation laws in physics (energy, momentum, charge)
- Determinant under row operations
- Parity in combinatorics
Meta-principle: "If something stays constant, make it your anchor."

QUICK DIAGNOSIS:
Trigger Patterns: "sum of", "total", "constant", "always", "for all", conservation laws, fixed sums
Diagnostic Questions:
1. Is there a quantity that remains constant throughout?
2. Can I express the problem in terms of this invariant?
3. Does tracking the invariant simplify the algebra?
MVC Template: "The seed is Invariance—[name the invariant] stays constant while [other things vary]. The brute path ignores this and tracks all moving parts. The pivot is to anchor the solution around the invariant."

---

2. SYMMETRY
Seed: The problem looks the same from multiple perspectives.
Elegance: Exploit the symmetry to reduce degrees of freedom.
Examples:
- Reflection, rotation, translation symmetry in geometry
- Permutation symmetry in algebra (symmetric polynomials)
- Even/odd function properties in calculus
- Exchange symmetry in physics (identical particles)
Meta-principle: "If the problem has symmetry, the solution inherits it."

QUICK DIAGNOSIS:
Trigger Patterns: Identical/interchangeable elements, "permutation", "rotation", "reflection", regular geometric objects
Diagnostic Questions:
1. If I swap/rotate/reflect elements, does the problem stay the same?
2. Are there variables that appear symmetrically?
3. Can I reduce the problem by exploiting this symmetry?

---

3. DUALITY
Seed: Two perspectives that are mirror images of each other.
Elegance: Solve in one domain, then translate back.
Examples:
- Geometry ↔ Algebra (coordinate vs. synthetic geometry)
- Time domain ↔ Frequency domain (Fourier transforms)
- Direct ↔ Inverse problems
- Primal ↔ Dual in optimization
Meta-principle: "If stuck in one language, switch to the dual."

QUICK DIAGNOSIS:
Trigger Patterns: Problem can be stated in two equivalent ways, Geometry ↔ Algebra formulations, Direct ↔ Inverse problems
Diagnostic Questions:
1. Can this be restated in another language/domain?
2. Is there a dual formulation that's simpler?

---

4. HIDDEN STRUCTURE (Pattern Recognition)
Seed: The problem has a disguised form of a known structure.
Elegance: Recognize the pattern, then apply the template.
Examples:
- Telescoping series (hidden cancellation)
- Quadratic disguised as linear (substitution u = x²)
- Recurrence relations (Fibonacci, geometric progressions)
- Self-similar structures (fractals, recursion)
Meta-principle: "If it looks unfamiliar, it's probably something familiar in disguise."

QUICK DIAGNOSIS:
Trigger Patterns: Problem looks unfamiliar but "feels" like something known, complex expression that might simplify, specific numbers that factor nicely
Diagnostic Questions:
1. Does this resemble a known pattern (telescoping, factorization, recursion)?
2. Are the given numbers/coefficients chosen for a reason?

========================================================
CATEGORY 2: TRANSFORMATION THINKING
"Change the representation to reveal simplicity."
========================================================

5. SUBSTITUTION / CHANGE OF VARIABLES
Seed: The problem is messy in one coordinate system, clean in another.
Elegance: Find the "right" variable that simplifies everything.
Examples:
- Trigonometric substitution in integrals
- u-substitution in differential equations
- Polar coordinates for circular symmetry
- Homogeneous coordinates in projective geometry
Meta-principle: "The right coordinate system makes the problem trivial."

QUICK DIAGNOSIS:
Trigger Patterns: Messy algebra with repeated expressions, square roots/radicals, trigonometric complexity
Diagnostic Questions:
1. Is there a repeated subexpression I can substitute?
2. Would a different variable make this cleaner?

---

6. LINEARIZATION / SIMPLIFICATION
Seed: A complex relationship can be approximated or transformed into a simpler one.
Elegance: Work in the simplified space, then lift the solution back.
Examples:
- Logarithms turning products into sums [Commentary #2]
- Taylor series approximations
- Small-angle approximations in physics
- Dimension reduction in high-dimensional problems
Meta-principle: "If nonlinear, find the linear core."

QUICK DIAGNOSIS:
Trigger Patterns: Nonlinear problem (products, exponentials), becomes linear after transformation, logarithms mentioned
Diagnostic Questions:
1. Can I take logs to turn products into sums?
2. Does linearization apply?

---

7. NORMALIZATION / SCALING
Seed: The problem has irrelevant scale or units.
Elegance: Absorb constants, work with dimensionless quantities.
Examples:
- Dividing by the leading coefficient
- Dimensional analysis in physics
- Normalizing probabilities to sum to 1
- Working with ratios instead of absolute values
Meta-principle: "Remove the clutter; keep only what matters."

QUICK DIAGNOSIS:
Trigger Patterns: Arbitrary constants/units, coefficients that obscure structure, "without loss of generality, let..."
Diagnostic Questions:
1. Can I absorb constants to simplify?
2. Would working with dimensionless quantities help?

---

8. TRANSLATION BETWEEN DOMAINS
Seed: The problem is stated in one domain but easier to solve in another.
Elegance: Move to the natural domain, solve, then translate back.
Examples:
- Algebra ↔ Geometry [Commentary #1: AP ↔ polygon angles]
- Discrete ↔ Continuous (sums ↔ integrals)
- Combinatorics ↔ Generating functions
- Graph theory ↔ Matrix algebra
Meta-principle: "If the language is wrong, speak a different one."

QUICK DIAGNOSIS:
Trigger Patterns: Problem stated in one language (algebra/geometry/combinatorics), solution clearer in another domain, "translate", "interpret", "view as"
Diagnostic Questions:
1. Is this algebraic but easier geometrically (or vice versa)?
2. Can I map this to a different domain where tools are better?

========================================================
CATEGORY 3: CONSTRAINT EXPLOITATION
"The restrictions are the solution."
========================================================

9. DOMAIN CONSTRAINTS / BOUNDS
Seed: Not all algebraic solutions are physically/geometrically valid.
Elegance: Filter candidates using domain reality before calculating.
Examples:
- Angles < 180° [Commentary #1]
- Time > 0 in kinematics
- Probabilities ∈ [0,1]
- Integer solutions only (Diophantine equations)
Meta-principle: "Algebra generates candidates; domain selects the winner."

QUICK DIAGNOSIS:
Trigger Patterns: Algebraic solutions exist but reality filters them, "valid", "physical", "geometrically possible", context clues (angles < 180°, time > 0)
Diagnostic Questions:
1. Do all algebraic roots make physical/geometric sense?
2. What domain constraints apply (positivity, boundedness, integrality)?
3. Can I filter solutions using domain checks BEFORE calculating?

---

10. INEQUALITY CONSTRAINTS
Seed: The problem isn't asking for exact values but for ranges or bounds.
Elegance: Use inequalities (AM-GM, Cauchy-Schwarz, triangle inequality) instead of equations.
Examples:
- Optimization problems (max/min under constraints)
- Bounding error terms
- Feasibility regions in linear programming
- Uncertainty principles in physics
Meta-principle: "Sometimes you don't need the answer—just its bounds."

QUICK DIAGNOSIS:
Trigger Patterns: "maximum", "minimum", "at least", "at most", inequalities appear, feasibility regions
Diagnostic Questions:
1. Is this asking for bounds rather than exact values?
2. Can I use a standard inequality (AM-GM, Cauchy-Schwarz)?

---

11. UNIQUENESS / EXISTENCE CONDITIONS
Seed: The problem asks "when does a solution exist?" or "is the solution unique?"
Elegance: Derive necessary and sufficient conditions instead of computing the solution.
Examples:
- Discriminant analysis (quadratic roots)
- Rank conditions (linear systems)
- Fixed-point theorems
- Constructibility in geometry (compass and straightedge)
Meta-principle: "Existence precedes computation."

QUICK DIAGNOSIS:
Trigger Patterns: "does there exist...", "find conditions for...", "necessary and sufficient conditions", discriminant, rank conditions
Diagnostic Questions:
1. Is this asking when a solution exists, not what it is?
2. Can I derive necessary conditions first?

---

12. EXTREMAL PRINCIPLES
Seed: Among all possibilities, the solution is the one that maximizes or minimizes something.
Elegance: Use calculus, inequalities, or variational methods to find extrema.
Examples:
- Fermat's principle (light takes shortest path)
- Least action in physics
- Max flow / min cut in networks
- Isoperimetric problems (max area for given perimeter)
Meta-principle: "Nature optimizes; so should your solution."

QUICK DIAGNOSIS:
Trigger Patterns: Problem asks for minimum/maximum, physics context (shortest path, least action), optimization language
Diagnostic Questions:
1. Is the solution characterized by being maximal/minimal?
2. Can I use calculus (derivatives = 0) or variational methods?

========================================================
CATEGORY 4: COUNTING & EXTREMIZATION
"Structure the chaos."
========================================================

13. COMBINATORIAL PRINCIPLES
Seed: Count configurations systematically instead of enumerating them.
Elegance: Use bijections, generating functions, or inclusion-exclusion.
Examples:
- Pigeonhole principle
- Double counting (same quantity, two ways)
- Stars and bars (distributing indistinguishable objects)
- Generating functions (encode sequences)
Meta-principle: "If you can't enumerate, structure the count."

QUICK DIAGNOSIS:
Trigger Patterns: Counting configurations/arrangements/selections, "in how many ways...", permutations/combinations
Diagnostic Questions:
1. Am I counting systematically or enumerating blindly?
2. Is there a bijection to a simpler counting problem?
3. Can I use generating functions, stars-and-bars, inclusion-exclusion?

---

14. PARITY / MODULARITY
Seed: Tracking residues (mod 2, mod 3, etc.) is easier than tracking exact values.
Elegance: Work modulo some number to simplify or eliminate cases.
Examples:
- Odd/even arguments
- Divisibility tests
- Fermat's Little Theorem (mod p)
- Coloring arguments (chessboard problems)
Meta-principle: "Sometimes the remainder tells the whole story."

QUICK DIAGNOSIS:
Trigger Patterns: Problem involves even/odd, divisibility, coloring arguments, residues modulo n
Diagnostic Questions:
1. Can I track parity instead of exact values?
2. Does working modulo some number simplify the problem?

---

15. BIJECTION / CORRESPONDENCE
Seed: Two seemingly different problems count the same thing.
Elegance: Solve the easier one, then map back.
Examples:
- Pairing combinatorial objects
- Catalan numbers (parenthesizations ↔ lattice paths)
- Dual graphs (faces ↔ vertices)
- Change of variables (bijective map between domains)
Meta-principle: "If two problems are isomorphic, solve the simpler one."

QUICK DIAGNOSIS:
Trigger Patterns: Two problems that "count the same thing", pairing/matching, "this is equivalent to..."
Diagnostic Questions:
1. Can I map this problem to a simpler one?
2. Is there a bijection between objects in two different settings?

========================================================
CATEGORY 5: META-REASONING
"Think about the problem structure itself."
========================================================

16. REVERSE ENGINEERING
Seed: The answer is given; find the question.
Elegance: Work backward from the desired conclusion to derive constraints.
Examples:
- Commentary #2 (given terms, find GP parameters)
- Proof by assuming the result and deriving necessity
- Constructing problems from solutions (problem design)
- Cryptanalysis (given ciphertext, find the key)
Meta-principle: "When the answer is given, the problem is to find the question."

QUICK DIAGNOSIS:
Trigger Patterns: "given the result, find the conditions", problem specifies outputs asks for inputs, "find necessary and sufficient conditions that..."
Diagnostic Questions:
1. Am I working backward from the desired conclusion?
2. Are the given values carefully chosen (not arbitrary)?
3. Does the problem structure suggest it was designed from the answer?

---

17. DEGREES OF FREEDOM ANALYSIS
Seed: Count independent variables vs. constraints.
Elegance: Overdetermined (DOF=0) → unique; underdetermined (DOF>0) → infinite; over-constrained → may have no solution.
Examples:
- System of equations (n equations, m unknowns)
- Dimension counting in geometry
- Rank-nullity theorem
- Configuration spaces in mechanics
Meta-principle: "Count constraints before solving."

QUICK DIAGNOSIS:
Trigger Patterns: System of equations with multiple variables, "overdetermined", "underdetermined", counting constraints vs. unknowns
Diagnostic Questions:
1. How many true unknowns vs. how many constraints?
2. Is the system over/under/exactly determined?
3. Which equations are independent vs. redundant?

---

18. RECURSION / INDUCTION
Seed: The solution for n depends on the solution for n-1.
Elegance: Establish base case + inductive step instead of solving all cases.
Examples:
- Mathematical induction
- Recurrence relations (Fibonacci, Tower of Hanoi)
- Dynamic programming
- Proof by strong induction
Meta-principle: "Solve for one step; repeat to infinity."

QUICK DIAGNOSIS:
Trigger Patterns: Solution for n depends on solution for n-1, "prove for all n", recurrence relations
Diagnostic Questions:
1. Can I express F(n) in terms of F(n-1)?
2. Is this provable by induction (base case + inductive step)?

---

19. PIVOTING / ELIMINATION
Seed: Reduce the problem by eliminating variables systematically.
Elegance: Remove degrees of freedom one at a time until trivial.
Examples:
- Gaussian elimination
- Substitution in simultaneous equations
- Lagrange multipliers (eliminate constraints)
- Variable elimination in logic
Meta-principle: "Simplify by subtraction, not addition."

QUICK DIAGNOSIS:
Trigger Patterns: System of equations that can be simplified, variables that can be eliminated systematically, Gaussian elimination, substitution cascades
Diagnostic Questions:
1. Can I eliminate variables one at a time?
2. Does forming ratios eliminate unwanted parameters?

---

20. ANALOGY / TRANSFER
Seed: This problem is structurally identical to a known one.
Elegance: Apply the solution template from the analogous problem.
Examples:
- Electric circuits ↔ fluid flow ↔ heat conduction (same differential equations)
- Harmonic oscillator (pendulum, LC circuit, quantum well)
- Random walks ↔ diffusion
- Optimization in different domains (same Lagrangian structure)
Meta-principle: "If you've solved it once, you've solved it everywhere."

QUICK DIAGNOSIS:
Trigger Patterns: "This reminds me of...", structural similarity to known problems, same differential equation in different contexts
Diagnostic Questions:
1. Have I seen this structure before in a different domain?
2. Can I apply a solution template from an analogous problem?

THE ULTIMATE SHORTCUT: THE "5-SECOND HEURISTIC"
========================================================

"What is the problem REALLY asking?"
- If it's asking "does X exist?" → Existence/Uniqueness (#11)
- If it's asking "find the maximum/minimum" → Extremal Principles (#12)
- If it's asking "count configurations" → Combinatorial Principles (#13)
- If it's giving you specific numbers → Reverse Engineering (#16) or Hidden Structure (#4)
- If it has algebraic + geometric language → Domain Translation (#8) or Domain Constraints (#9)
- If something stays the same → Invariance (#1)
- If things look identical → Symmetry (#2)

========================================================
ADVANCED TIPS: MULTI-ARCHETYPE PROBLEMS
========================================================

Primary + Secondary Pattern:
- Primary: The main structural insight (goes in Seed)
- Secondary: Alternative approaches or supporting techniques (goes in Connections)

Example:
- Primary: Domain Constraints (angles < 180°)
- Secondary: Substitution (let u = n-2), Normalization (divide by 5°), Degrees of Freedom (overdetermined system)

Parallel Paths (some problems can be solved via completely different archetypes):
Example (GP problem):
- Path 1: Linearization (take logs, convert GP → AP)
- Path 2: Reverse Engineering (work backward from given ratios)
- Path 3: Symmetry (focus on index differences, not absolute positions)

In your MVC: "The primary seed is Reverse Engineering, but alternative paths exist via Linearization (log transformation) or Symmetry (index differences). All converge to the same constraints."

========================================================
COMMON PITFALLS IN ARCHETYPE IDENTIFICATION
========================================================

Pitfall 1: Confusing Substitution with Translation
- Substitution: Change of variables WITHIN the same domain (let u = x²)
- Translation: Moving BETWEEN domains (algebra → geometry)

Pitfall 2: Missing Hidden Structure
- Given numbers like 27, 8, 12 are NEVER arbitrary—they encode the solution
- Always factor/analyze special values

Pitfall 3: Overlooking Domain Constraints
- Algebra always generates candidates; physical/geometric reality filters
- ALWAYS ask: "Do all roots make sense in context?"

Pitfall 4: Treating Symmetry as Just "Pretty"
- Symmetry isn't decorative—it REDUCES DEGREES OF FREEDOM
- Exploit it to collapse cases

Pitfall 5: Ignoring Degrees of Freedom
- Count constraints vs. unknowns BEFORE solving
- Overdetermined → expect rejection; underdetermined → expect parameters
